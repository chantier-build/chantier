Output: Five `core/schemas/*.json` files, each a valid JSON Schema draft-07 document constrained to the keyword subset the jq validator supports (`type / required / properties / additionalProperties / pattern / enum / items`). Each file uses `additionalProperties: true` at the top level (D-05 hybrid mode); the validator emits a warning — not an error — for unknown keys.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| schema files → validate-task (future plan 02-05) | schemas are author-controlled and version-controlled; validate-task reads them at runtime |
| schema enum values → NFR-001 harness deny-list | skill.json's harness_adapters enum is the sole legitimate occurrence of harness names in core/; the self-test in plan 02-03 must respect this carve-out |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-DOS | DoS | jq schema validator | mitigate | Per RESEARCH §Security Domain row "Schema-validator regex DoS" — all schemas live under core/schemas/ and are author-controlled, not user-supplied. ADR 0002 documents this assumption. No regex with catastrophic-backtracking risk allowed in `pattern` fields. The patterns chosen here (slug, ISO date, semver) are linear-time. |
| T-02-02-DRIFT | Tampering | inline ADR 0002 schemas vs core/schemas/*.json (future) | mitigate | Plan 02-06 must include a check (or document a self-test step) verifying the ADR's inline ```json blocks match the file content byte-for-byte. Cross-referenced in plan 02-06. |
| T-02-02-INJ | Tampering | regex pattern fields in schemas | mitigate | All `pattern` regexes are anchored (`^...$`) and use linear-time character classes. No `.*` greedy + nested quantifier patterns. |
| T-02-02-NFR | Repudiation | harness identifier leakage | mitigate | skill.json explicitly documents the NFR-001 carve-out in its `description` field so the self-test grep in plan 02-03 knows to exclude this single file (the binary itself MUST remain harness-free). |
| T-02-02-FORWARD | InfoDisclosure | additionalProperties: true permits unknown keys | accept | This is D-05 hybrid mode by design — unknown forward-compat keys generate a warning, not an error. The trade-off is documented in CONTEXT.md and will be reified in ADR 0002. |
</threat_model>
