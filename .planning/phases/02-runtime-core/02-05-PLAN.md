    1. **`validate_against_schema()`** — per RESEARCH Pattern 5 lines 487–558 verbatim. Takes two args: `target` (path to JSON file) and `schema` (path to schema file). Invokes `jq -e -n --slurpfile data --slurpfile schema` with the embedded validator filter. Emits one violation per line on stderr prefixed `schema violation: `. Returns 0 if no violations, 1 if any. Implements only the supported keyword subset (`type`, `required`, `properties`, `additionalProperties`, `pattern`, `enum`, `items`). Anything else in the schema is silently ignored — this is the documented contract per RESEARCH lines 473–480 and will be codified in ADR 0002 (plan 02-06).

    2. **`extract_frontmatter_as_json()`** — per RESEARCH §"Code Examples > Verifying a frontmatter file" lines 737–762 verbatim. Takes one arg `file`, uses awk to extract everything between the first two `---` delimiters, pipes to jq with a `reduce` filter that converts each `key: value` line into a JSON object. Handles only the frontmatter subset profile (top-level scalars + simple lists) — nested maps not supported, which is the ADR 0002-documented constraint.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PLAN.md → validate_task | task-block YAML is awk-extracted; values flow into shell vars, never eval'd |
| SKILL.md → validate_task gate 4 | declared state_writes paths must be canonicalized before write-containment check |
| skill body files → gate 4 grep | grep is read-only and pattern is literal regex; no shell expansion of file contents |
| user-supplied $name → new_project heredocs | $name is single-quoted-shell-arg; only interpolated inside heredocs via $name, never eval'd |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
