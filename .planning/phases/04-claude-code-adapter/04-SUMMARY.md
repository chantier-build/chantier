## Surface 3 propagation discovery (Phase 4 plan 03 Rule-1 auto-fix)

Plan 02 shipped the adapter with the dossier-staging side correct (Surface 2) but missed the dossier→state_writes propagation needed for Surface 3 (validate-task gates 1-2). The Phase 4 e2e test in plan 03 surfaced this within seconds of the first run: the skill ran clean and emitted output.md+output.json in the dossier (its `$PWD`), but `chantier validate-task` looked in `.planning/phases/<phase>/tasks/<task>/` and found nothing. The 12-line fix (copy all plain files from dossier root to TASK_DIR, except `inputs.yml`, `env.sh`, `subagent.transcript.log`) honors D-08 preservation via `cp` semantics and works uniformly across all four Phase 3 skills (none of which were modified). This is the "tested-only" claim from Phase 3 D-14 working as intended: the e2e test catches the gap the smoke-test missed.

## Handoff Notes for Phase 5 (Dogfood E2E)

Phase 5 inherits:

1. **The adapter is invocable and proven end-to-end.** `adapters/claude-code/run-task.sh <task-id>` works from inside any git worktree against any Phase 3 skill that follows the `TASK_DIR="${PWD}"` + `output.md/output.json` emission pattern (all four Phase 3 skills do, by design).
2. **`chantier state append` + `chantier validate-task` + the adapter compose correctly.** The three-event bracketing (task.started / skill.completed / task.completed) lands one row each in STATE.md, in the right order, via the binary's mkdir-mutex (NFR-003).
3. **The bats suite is at 73/0.** Phase 5 should not regress this; the integration test in `tests/e2e/` (which Phase 5 ships) is additive.
4. **The four Phase 3 skills are testably honest.** `test-driven-development` is exercised directly through the adapter; the other three (`using-git-worktrees`, `requesting-code-review`, `subagent-driven-development`) follow the same canonical run.sh shape per the Phase 3 D-14 + D-17 mechanical extension criterion. Phase 5 may exercise the full matrix (one e2e per skill through the adapter) if appropriate.
5. **The Surface 3 propagation block in the adapter is a contract, not an accident.** Future skills must continue to write `output.md` + `output.json` (and any artifacts) in their `$PWD` (the dossier). The adapter copies these to TASK_DIR. The Phase 4 plan 02 OQ recommended this behavior implicitly via D-08 preservation; plan 03 made it explicit.

### Deferred items (carried into Phase 5+ backlog)

- **Concurrency lock for parallel `run-task.sh` invocations on the same task ID.** Not designed for v0.1; mkdir-mutex on `.chantier/dossiers/<task>/.lock` is the natural extension if Phase 5 dogfood surfaces a need.
- **Transcript capture inspection on real-claude path.** Plan 02 added the redirect; v0.2+ should inspect verbosity and gate behind `CHANTIER_TRANSCRIPT=1` if logs grow unbounded.
- **`chantier task-lookup <id>` subcommand.** Plan 02 inlined the lookup in the adapter; if a second adapter ships in v0.2.0, this becomes a candidate refactor (single source of truth for PLAN.md task resolution).
- **Second-adapter audit symmetry.** When `adapters/cursor/` (or similar) ships, `core/tests/adapter_isolation.bats` should add a parallel case-arm matching the new directory. The audit's current shape is "additive by case arm"; no structural change needed.
- **`upstream/` symlink staging on `depends_on`.** Plan 02 emits the empty directory only; multi-task plans where one task's output feeds another's `upstream/` need `extract_task_field "$TASK_ID" depends_on block-dash` + symlink loop. Mechanical extension.
