## Findings / Gaps for Phase 5

None of the items below block Phase 5; they are forward-looking observations surfaced by this audit that the Phase 5 (Dogfood E2E) planner may want to internalize.

### F1 — Surface 3 propagation is a load-bearing contract, not yet ADR-documented

**Severity:** INFO (not a blocker; documented in 04-03-SUMMARY.md and 04-SUMMARY.md Handoff Notes §5)

**Observation:** Plan 04-03 surfaced a real bug in plan 04-02's adapter: the skill writes `output.md` + `output.json` in its `$PWD` (the dossier) per Phase 3 D-04, but `chantier validate-task` looks under `.planning/phases/<phase>/tasks/<task>/`. The plan 03 Rule-1 fix (12-line propagation loop in `run-task.sh:213-219`) copies dossier outputs to `TASK_DIR` between dispatch and validate-task. This is now the canonical adapter contract for all future adapters — but it is documented only in 04-SUMMARY.md "Handoff Notes" and not in ADR 0001 or ADR 0002.

**Phase 5 action:** Consider whether ADR 0001 Surface 3 should be amended (or a new ADR drafted) to explicitly state "the adapter is responsible for propagating skill outputs from the Surface 2 dossier to the Surface 3 `state_writes` path before invoking `chantier validate-task`." Currently a future adapter author could re-introduce the same bug.

### F2 — Real-claude dispatch is unproven

**Severity:** INFO (deferred to v0.2+ per CONTEXT.md "Deferred Ideas" — out of Phase 4 scope by design)

**Observation:** SC#2's end-to-end proof flows through the deterministic `CHANTIER_CLAUDE_BIN` stub. The real `claude -p` binary is never invoked in CI (NFR-004 forbids network). The dispatch prompt's wording (`run-task.sh:165-181`), the transcript-capture redirect (`run-task.sh:188`), and the real-binary indirection (`run-task.sh:77-79`, `190`) are all code-verified but not behavior-verified against a live Claude session.

**Phase 5 action:** Phase 5 may layer a real-claude e2e variant gated behind an env var (e.g., `CHANTIER_E2E_REAL_CLAUDE=1`) for release-time validation, separate from the CI default. The full-matrix exercise (all four Phase 3 skills through the adapter, not just `test-driven-development`) also falls naturally into Phase 5's dogfood scope.

### F3 — `upstream/` symlink staging is a no-op in the current adapter

**Severity:** INFO (deferred to Phase 5 per 04-SUMMARY.md Handoff Notes §"Deferred items")

**Observation:** The adapter creates `$DOSSIER/upstream/` (run-task.sh:118) but never populates it. The Phase 4 e2e fixture has no `depends_on`, so this is correct for v0.1. Multi-task plans where one task's output feeds another's `upstream/` need an `extract_task_field "$TASK_ID" depends_on block-dash "$PLAN_PATH"` invocation + symlink loop — mechanical extension, no architectural change.

**Phase 5 action:** If the Phase 5 dogfood exercises a multi-task plan, the adapter will need the `upstream/` population block. Symmetric to the existing `state_reads` symlink loop (run-task.sh:139-143).

### F4 — Strict-vs-lax worktree validation chose lax for v0.1

**Severity:** INFO (D-05 RESEARCH A5 lax disposition — deliberate trade-off)

**Observation:** `run-task.sh:85` uses `git rev-parse --show-toplevel` only, which accepts both linked worktrees and the main checkout. CONTEXT.md says ROADMAP SC#3 ("in a worktree") would prefer strict semantics ("must be in a `git worktree add`-created worktree"), but RESEARCH A5 recommended lax for v0.1 dev-loop ergonomics. The Phase 4 e2e satisfies SC#3 strictly (the test uses `git worktree add`), but the adapter does not enforce it.

**Phase 5 action:** If Phase 5 dogfood surfaces a real bug from running in the main checkout (e.g., race on `.chantier/dossiers/` across two terminals on the same repo), consider tightening to strict. Otherwise, leave as-is.
